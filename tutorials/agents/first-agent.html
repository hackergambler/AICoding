Build Your First Agent

Learn Build Your First Agent in the AI Agents course. Practical Python code and real-world examples.

Your first agent architecture

For a first serious build, keep the architecture brutally simple: one model, a small tool set, a max-step loop, and logs for every decision. That is enough to build something useful without creating a debugging nightmare.

  • Model decides whether to answer or call a tool
  • Tool registry maps names to real Python functions
  • Execution loop runs until the model stops or the step limit is hit
  • Trace log records what happened at each step

1. Start with narrow tools

The fastest way to make a first agent unreliable is to give it tools that are vague, overlapping, or overly powerful. Start with tools that do one clear thing and return compact results.

Python - narrow tools are easier to control
def search_web(query: str) -> str:
    """Search the web for public facts and return the top snippets."""
    return "Top snippets for: " + query

def calculator(expression: str) -> str:
    """Evaluate a safe arithmetic expression."""
    return str(eval(expression, {"__builtins__": {}}, {}))

These are imperfect toy tools, but they are easy to inspect and easy to reason about.

2. Run a complete tool-calling loop

The core idea is simple: ask the model, inspect tool calls, run them, append the results, and ask again. The code below is intentionally explicit so you can see the loop clearly.

Python - first complete agent loop
import json
from openai import OpenAI

client = OpenAI()

TOOLS = [
    {"type": "function", "function": {"name": "search_web", "description": "Search public web facts.", "parameters": {"type": "object", "properties": {"query": {"type": "string"}}, "required": ["query"]}}},
    {"type": "function", "function": {"name": "calculator", "description": "Evaluate arithmetic expressions.", "parameters": {"type": "object", "properties": {"expression": {"type": "string"}}, "required": ["expression"]}}}
]

TOOL_FNS = {"search_web": search_web, "calculator": calculator}

def run_agent(user_query, max_steps=5):
    messages = [
        {"role": "system", "content": "You are a research assistant. Use tools when needed and stop when you can answer clearly."},
        {"role": "user", "content": user_query}
    ]

    for step in range(max_steps):
        response = client.chat.completions.create(model="gpt-4o-mini", messages=messages, tools=TOOLS)
        msg = response.choices[0].message
        messages.append(msg)

        if not msg.tool_calls:
            return msg.content

        for tc in msg.tool_calls:
            args = json.loads(tc.function.arguments)
            result = TOOL_FNS[tc.function.name](**args)
            messages.append({
                "role": "tool",
                "tool_call_id": tc.id,
                "content": result
            })

    return "Stopped after max steps."

3. Add traces so you can trust the loop

Agents feel impressive when they work and frustrating when they fail. Add a tiny trace object now so every run records the question, tool calls, arguments, and final answer.

Python - lightweight trace logging
trace = []

trace.append({"step": step, "tool": tc.function.name, "args": args, "result_preview": result[:120]})

for event in trace:
    print(event)
💡 Make logs human-readable

You do not need full observability infrastructure on day one. Even a simple list of step events is enough to reveal bad tool choices, argument errors, or loops.

4. Failure patterns you should expect

  1. Infinite or wasteful loops — fix with max steps and better stopping prompts
  2. Wrong tool selected — your tool descriptions overlap too much
  3. Tool output is too verbose — return compact, relevant results only
  4. Unsafe execution — validation is missing before the tool runs

Use Debug Lab for runtime triage and Configuration Debugging when the environment or SDK is the real issue.

5. Expand the first agent the right way

Do not jump straight into ten tools and memory. Add one layer at a time:

  • Function Calling Basics — tighter tool contract design
  • ReAct Loop — better reasoning/action pattern
  • Plan and Execute — multi-step tasks with stronger structure
  • Dev Agent project — apply the whole stack in a larger build
