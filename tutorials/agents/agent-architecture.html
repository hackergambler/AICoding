Production Agent Architecture

Learn how to structure a production agent as a system with state, tools, planning, verification, and observability — not just a long prompt.

Objective & architecture principle

A production agent is not a single LLM call. It is a system made of state, policies, tools, memory, execution limits, and review logic. Strong architecture reduces randomness by making every step explicit.

  • Planner proposes the next step or tool.
  • Executor performs the step in a bounded environment.
  • Verifier checks whether the result is usable, safe, or complete.
  • State store records what happened so the agent can continue or recover.
Python — minimal planner / executor / verifier loop
from dataclasses import dataclass, field
from typing import List, Dict

@dataclass
class AgentState:
    goal: str
    observations: List[str] = field(default_factory=list)
    actions: List[str] = field(default_factory=list)
    done: bool = False


def planner(state: AgentState) -> str:
    if not state.observations:
        return "search_docs"
    if any("missing" in obs.lower() for obs in state.observations):
        return "ask_user_for_detail"
    return "draft_answer"


def executor(action: str, state: AgentState) -> str:
    tools: Dict[str, str] = {
        "search_docs": "Found 3 relevant internal docs.",
        "ask_user_for_detail": "Need invoice ID to continue.",
        "draft_answer": "Prepared a cited response for review."
    }
    state.actions.append(action)
    return tools[action]


def verifier(state: AgentState, result: str) -> bool:
    if "Need invoice ID" in result:
        return True
    if "cited response" in result:
        state.done = True
        return True
    return False

Architecture layers that matter

  1. Control layer: rules for how the agent decides, loops, retries, or escalates.
  2. Tool layer: the narrow interfaces the model can call. Each tool should have a stable schema and bounded permissions.
  3. State layer: working memory, task status, intermediate outputs, and audit logs.
  4. Safety layer: limits on actions, review steps, policy checks, and sandboxing for risky operations.
  5. Observability layer: traces, cost metrics, action history, and error buckets.
💡 Architecture rule of thumb

If you cannot explain where the agent stores state, where tool permissions are defined, and how a bad step is detected, you do not yet have a production architecture — you have a prompt workflow.

Python — bounded execution policy
MAX_STEPS = 8
ALLOWED_TOOLS = {"search_docs", "ask_user_for_detail", "draft_answer"}

for step in range(MAX_STEPS):
    action = planner(state)
    if action not in ALLOWED_TOOLS:
        raise ValueError(f"Blocked tool: {action}")

    result = executor(action, state)
    state.observations.append(result)

    if verifier(state, result):
        if state.done:
            break
else:
    raise RuntimeError("Agent exceeded allowed execution steps")

Common production failures

  • No explicit state: the whole workflow depends on the model remembering prior steps from raw text.
  • Unlimited loops: the agent keeps trying tools without a max-step policy or escalation rule.
  • Unverified outputs: tool results and final answers are accepted without checks.
  • Weak tool contracts: the agent calls tools with ambiguous parameters or broad permissions.

Hands-on exercise

Design an agent that triages support tickets. Give it a planner, two tools, a verifier, and a max-step rule. Then simulate three failure cases: missing ticket ID, wrong tool choice, and repeated loop. Decide when the system should escalate to a human instead of continuing autonomously.

Real-world use case

Operations copilots, research agents, and developer assistants all need architecture more than extra prompting. When the system has explicit control flow, stable state, and narrow tool boundaries, the team can debug and improve it. Without that structure, every failure looks like “the model did something weird.”
