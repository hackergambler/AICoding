Unit Testing Agents

Unit test agent decision logic with model stubs, fake tools, and deterministic fixtures so you can validate tool choice, arguments, and stop conditions before end-to-end runs.

What Unit Tests Protect in Agent Systems

End-to-end demos hide too much. Unit tests let you pin down the parts of an agent that should behave predictably: tool routing, argument shaping, refusal behavior, guardrails, and stop conditions. They are the fastest way to keep an agent from regressing every time you touch prompts or tool schemas.

Testing mindset

Do not try to prove that a model is universally correct. Instead, prove that your orchestration code reacts correctly to representative model outputs and tool results.

A Useful Test Pyramid

  • Pure helper tests: parsers, schema validators, budget checks, patch filters, retry policies.
  • Decision tests: given a model response, does the agent choose the right tool and arguments?
  • Tool contract tests: does each tool return stable structured data and clear errors?
  • Mini workflow tests: one or two steps with fake tools, no network, deterministic timing.

Stub the Model, Fake the Tools

The cleanest unit tests replace the model with a deterministic stub and replace external tools with tiny fake implementations. That lets you assert on what the agent tried to do instead of depending on real APIs during every test run.

Python - deterministic unit test for tool routing
class StubModel:
    def __init__(self, scripted_response):
        self.scripted_response = scripted_response

    def respond(self, messages):
        return self.scripted_response


class FakeSearchTool:
    def __init__(self):
        self.calls = []

    def __call__(self, query):
        self.calls.append({"query": query})
        return {"results": ["doc-1", "doc-2"]}


def test_agent_routes_to_search_tool():
    model = StubModel({
        "action": "tool_call",
        "tool_name": "search_docs",
        "arguments": {"query": "refund policy enterprise"}
    })
    search_tool = FakeSearchTool()

    result = run_agent_turn(
        model=model,
        tools={"search_docs": search_tool},
        user_message="Find the enterprise refund policy"
    )

    assert search_tool.calls == [{"query": "refund policy enterprise"}]
    assert result["status"] == "tool_executed"
    assert result["tool_name"] == "search_docs"

What to Assert

  1. Tool choice: did the agent select the expected class of action?
  2. Arguments: were required fields present, typed correctly, and normalized?
  3. Stop condition: did the agent stop after success instead of looping?
  4. Escalation: did the agent refuse or hand off when the policy required it?
Common mistake

Teams often unit test the exact wording of a long response. That creates brittle tests. Prefer structural assertions: selected tool, argument payload, policy branch, or presence of required fields in the summary.

Golden Traces Without Overfitting

Golden traces are useful when you want to preserve the shape of a good run. Save a compact expected trace with tool calls, checkpoint names, and termination reason. Avoid snapshotting every token, because that makes harmless wording changes look like failures.

Contract Tests for Tools

Every production tool should be tested independently from the agent. Validate success cases, validation errors, permission failures, timeouts, and empty-result behavior. When the tool contract is stable, agent tests become much simpler.

Build This Into the Capstones

  • Dev Agent — unit test patch-planning, path allow-lists, and test command selection.
  • Research Crew — verify citation extraction, source filtering, and editorial handoff.
  • Coding Agent — apply deterministic tests to repository mapping and patch planning.
  • Trajectory Evaluation — use unit tests to validate scoring helpers before grading full runs.
  • Debugging Lab — isolate flaky tool wrappers and replay failures locally.

Practice Exercise

Write three tests for a customer-support agent: one where it uses the refund-policy tool correctly, one where it refuses a password-reset request without verification, and one where the knowledge-base tool returns an empty result. Make each test assert the chosen branch and the stop condition.
