Project: Coding Agent

Design coding agents that inspect a repository, plan bounded edits, run tests, and escalate risky changes instead of blindly rewriting code.

Why Coding Agents Are Hard

A coding agent has to combine language reasoning with real software constraints: repository layout, file boundaries, test signals, shell commands, and rollback paths. That makes it much more brittle than a chatbot that only writes snippets in a text box.

Mental model

Treat a coding agent as a planning system with tools, not as “the model that somehow fixes code.” The model proposes, but tools and guards verify.

Core Loop

  1. Understand the task and repository shape.
  2. Choose a minimal edit plan.
  3. Apply the patch only to allowed files.
  4. Run narrow tests or checks.
  5. Summarize failures and either repair or escalate.
Python - coding agent control loop
def coding_agent(task, repo_map):
    history = []
    for attempt in range(3):
        plan = model_plan(task=task, repo_map=repo_map, history=history)
        patch_result = apply_patch(plan["patch"])
        test_result = run_tests(plan["tests"])

        history.append({
            "attempt": attempt + 1,
            "patch": patch_result,
            "tests": test_result,
        })

        if test_result["returncode"] == 0:
            return {"status": "success", "history": history}

        if plan.get("needs_human_review"):
            break

    return {"status": "escalate", "history": history}

Repository Understanding Before Editing

The first job is not code generation. It is choosing the right files. Good coding agents build a compact repository map, identify high-signal files, and capture a few invariants such as public API names, test commands, and directories that should never be touched.

  • Repo map: relative paths, file sizes, and top-level folders.
  • Critical files: package config, environment files, CI config, migrations, auth boundaries.
  • Acceptance contract: what counts as “done” for this task.

Patch Planning

Ask the model for a structured patch plan before it writes. The plan should name files to edit, tests to run, risks, and a stop condition. This single design choice makes debugging much easier because you can inspect intent before code changes happen.

Risk Controls That Matter

  • Path allow-lists keep the agent inside the workspace.
  • Command policies block dangerous shell actions by default.
  • Patch size caps stop massive rewrites on small tasks.
  • Human review gates protect migrations, auth, and public APIs.
Common failure pattern

The agent appears helpful because it keeps changing files, but every loop drifts farther from the real bug. When the failure summary stays vague after two attempts, stop and escalate instead of letting the model thrash.

How to Evaluate a Coding Agent

Do not grade only on “green tests.” Track first-pass success, average patch size, time-to-green, regression rate, and how often the agent requested human review appropriately.

Build This Into a Capstone

  • Project: Coding Agent — build the full end-to-end system.
  • Tool Schemas — design safe file and shell tools.
  • Unit Testing Agents — test agent decisions deterministically.
  • Stress Testing Agents — probe edge cases before broader rollout.
  • Debugging Lab — fix sandbox and execution issues.

Practice Exercise

Create a tiny repository with one failing test and one distracting unrelated file. Force your agent to produce a patch plan first. Measure whether it touches only the correct file, runs the smallest useful test command, and stops after success instead of continuing to “improve” the code.
