Trajectory Evaluation

Evaluate full agent trajectories with step-level rubrics for planning, tool choice, recovery, policy compliance, and efficiency instead of grading only the final answer.

Why Final-Answer Scoring Misses Agent Failures

An agent can sometimes reach the right answer for the wrong reasons: using an unsafe tool, taking ten useless steps, or recovering badly from a failure. Trajectory evaluation looks at the entire run so you can grade planning quality, execution discipline, and policy behavior, not just the final string returned to the user.

Mental model

For agents, the run itself is part of the product. A correct outcome reached through unsafe or wasteful behavior is still a weak trajectory.

What a Good Trace Should Record

  • User goal and any constraints or budgets.
  • Plan updates when the agent changes course.
  • Tool calls with arguments, outcomes, and latency.
  • Guardrail events such as blocked commands or refusals.
  • Termination reason — success, degrade, escalate, timeout, or budget stop.

Scoring Dimensions

  1. Task completion: did the run satisfy the real request?
  2. Tool quality: were the chosen tools appropriate and arguments well formed?
  3. Efficiency: how many unnecessary steps or token-heavy loops happened?
  4. Recovery behavior: did the agent react correctly after errors?
  5. Policy compliance: were risky actions blocked or escalated properly?
Python - simple trajectory scorer
def score_trajectory(trace):
    score = 0
    notes = []

    if trace["task_success"]:
        score += 4
    else:
        notes.append("task not completed")

    if trace["policy_violations"] == 0:
        score += 2
    else:
        notes.append("policy breach or blocked attempt")

    if trace["unnecessary_steps"] <= 1:
        score += 2
    else:
        notes.append("too much wandering")

    if trace["recovery_quality"] == "good":
        score += 2
    elif trace["recovery_quality"] == "bad":
        notes.append("poor recovery decision")

    return {"score": score, "notes": notes}

LLM Judges and Deterministic Metrics

Use deterministic metrics where possible: number of steps, invalid tool arguments, timeout count, policy blocks, and budget usage. Then add an LLM judge for qualitative dimensions such as whether the plan made sense or whether the final handoff summary was actionable. Combining both gives you better signal than either alone.

Do not overtrust the judge

LLM judges are useful, but they can be fooled by fluent traces. Always pair them with concrete metrics like tool validity, step count, latency, and policy events.

Trajectory Rubrics for Different Agent Types

  • Coding agents: file selection, patch size, test relevance, escalation timing.
  • Research agents: source quality, citation completeness, claim/evidence alignment.
  • Workflow agents: duplicate writes avoided, retries scoped properly, queue state preserved.
  • Personal assistants: correct calendar/email actions, memory hygiene, safe delegation.

Build This Into the Capstones

  • Research Crew — grade source selection and citation-backed conclusions.
  • Dev Agent — score patch planning, test choice, and stop conditions.
  • Error Recovery — feed recovery outcomes into the trace rubric.
  • Agent Benchmarks — compare trajectory scores across releases.
  • Data Debugging — inspect corrupted traces and missing event fields.

Practice Exercise

Collect ten agent traces from one capstone. Build a rubric with at least five dimensions, then compare your scores against plain final-answer pass/fail. Notice which low-quality runs would have slipped through if you only graded the end result.
