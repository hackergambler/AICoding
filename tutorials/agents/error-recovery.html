Agent Error Recovery Patterns

Design recovery ladders for agents so they retry transient failures, repair bad state, checkpoint progress, and escalate when autonomy is no longer safe.

Recovery Is a Control Policy, Not a Retry Loop

When an agent fails, the most expensive mistake is usually repeating the same step with the same bad assumptions. Strong recovery design asks a different question: what changed, what stayed valid, and what is the safest next action?

Mental model

Treat recovery as a ladder. Start with the cheapest safe action, but keep a clear point where the agent must re-plan, degrade capability, or hand the task to a human.

Failure Taxonomy

  • Transient infrastructure failures: rate limits, timeouts, flaky browsers, temporary database locks.
  • Tool contract failures: missing fields, invalid JSON, wrong argument types, stale resource identifiers.
  • State corruption: the agent wrote partial data, lost track of the active item, or mixed two tasks together.
  • Policy failures: the model tried to use an unsafe command, touched a blocked path, or exceeded a budget.
  • Reasoning failures: the answer is coherent, but the chosen plan is wrong for the task.

The Recovery Ladder

  1. Retry the same step only for clearly transient failures and only with capped attempts.
  2. Repair inputs or state if the failure came from missing context, invalid arguments, or stale identifiers.
  3. Re-plan when the original route no longer makes sense.
  4. Degrade gracefully by skipping a non-critical step, switching to a safer tool, or returning a partial result.
  5. Escalate when the failure touches policy, money, auth, customer-facing writes, or repeated uncertainty.
Common anti-pattern

Blindly increasing retries hides design problems. If the same tool call fails twice for non-transient reasons, the agent should stop, summarize the evidence, and choose a different path.

Recovery Controller

Python - recovery ladder with checkpoints
from dataclasses import dataclass
from typing import Any

@dataclass
class StepResult:
    ok: bool
    kind: str
    data: dict[str, Any] | None = None
    error: str | None = None

MAX_RETRIES = 2

def recoverable(kind: str) -> bool:
    return kind in {"timeout", "rate_limit", "temporary_unavailable"}


def run_step(step_name, fn, state, attempt=0):
    result = fn(state)
    if result.ok:
        state["checkpoints"][step_name] = result.data or {}
        return {"action": "continue", "state": state, "result": result}

    if recoverable(result.kind) and attempt < MAX_RETRIES:
        return {"action": "retry", "attempt": attempt + 1, "state": state}

    if result.kind in {"invalid_args", "missing_context", "stale_handle"}:
        repaired = repair_state(step_name, state, result)
        return {"action": "replan", "state": repaired, "reason": result.kind}

    if result.kind in {"unsafe_write", "auth_boundary", "budget_exceeded"}:
        return {"action": "escalate", "summary": summarize_failure(step_name, state, result)}

    return {"action": "degrade", "state": state, "fallback": "partial_result"}

Checkpointing and Idempotency

Recovery gets much easier when each step records enough structured state to resume safely. For workflow agents this might be the current item ID, completed actions, outbound message IDs, and a small explanation of why the next step exists. For coding agents it might be the selected files, patch hash, and tests already executed.

  • Checkpoint after side effects so a restarted run knows what already happened.
  • Use idempotency keys for email sends, ticket creation, or API writes that must not duplicate.
  • Store failure summaries in a machine-readable way so retries can use them.

Compensating Actions

Some failures happen after partial success. In that case the right move is not “retry harder,” but a compensating action: delete the temporary document, close the draft ticket, revert the risky patch, or mark the item for manual review.

Design rule

Every tool with write access should document both its forward action and its recovery action. If you can create something, ask how you cancel, roll back, or reconcile it.

How to Grade Recovery Quality

  • Retry precision: did the agent retry only transient failures?
  • State integrity: could the run resume without duplicating work?
  • Escalation quality: was the human handoff short, evidence-backed, and actionable?
  • User impact: did the system return a safe partial result instead of disappearing?

Build This Into the Capstones

  • Workflow Automation — add item-level retries, idempotency keys, and manual-review queues.
  • Personal Assistant — protect reminders, email drafts, and memory writes with recovery checkpoints.
  • Agent Monitoring — turn failures into visible buckets and dashboards.
  • Trajectory Evaluation — score whether the agent recovered in the right way, not just whether it eventually answered.
  • Configuration Debugging — trace environment and secret issues that look like reasoning failures.

Practice Exercise

Take a three-step workflow: fetch a task, transform data, and send a result. Inject one timeout, one invalid argument, and one permission failure. For each case, decide which rung of the recovery ladder should fire and what the run summary should look like for a human reviewer.
