LangGraph Basics

Learn how LangGraph turns agent loops into explicit nodes and state transitions so complex workflows become easier to debug and extend.

Objective & why graph-based agents help

Once an agent has more than one tool, more than one decision point, or more than one retry rule, plain chat history becomes a poor control system. LangGraph helps by representing the workflow as state + nodes + transitions. That gives you a more inspectable mental model than a free-form loop.

Python — minimal LangGraph state machine
from typing import TypedDict
from langgraph.graph import StateGraph, END

class SupportState(TypedDict):
    question: str
    docs_found: bool
    draft_ready: bool
    answer: str


def retrieve(state: SupportState):
    return {"docs_found": True}


def answer(state: SupportState):
    return {"draft_ready": True, "answer": "Use policy-12 and policy-18 in the final reply."}


def route(state: SupportState):
    if state["docs_found"]:
        return "answer"
    return END

workflow = StateGraph(SupportState)
workflow.add_node("retrieve", retrieve)
workflow.add_node("answer", answer)
workflow.set_entry_point("retrieve")
workflow.add_conditional_edges("retrieve", route, {"answer": "answer", END: END})
workflow.add_edge("answer", END)

graph = workflow.compile()
print(graph.invoke({"question": "How do invoice disputes work?", "docs_found": False, "draft_ready": False, "answer": ""}))

Core concepts

  1. State schema: define the fields every node can read or update.
  2. Nodes: each node performs one clear unit of work such as retrieval, tool use, or verification.
  3. Edges: transitions decide what happens next based on state.
  4. Termination: every graph needs explicit conditions for finishing or escalating.
💡 Keep nodes narrow

When a single node both retrieves documents, chooses tools, and drafts an answer, the graph becomes hard to reason about. Smaller nodes make traces easier to read and retries safer to implement.

Python — add a review node before final output
def review(state: SupportState):
    if "policy" not in state["answer"].lower():
        return {"answer": "Need a cited answer before returning to the user."}
    return state

workflow.add_node("review", review)
workflow.add_edge("answer", "review")
workflow.add_edge("review", END)

Failure patterns

  • State explosion: too many unrelated fields make every node fragile.
  • Implicit transitions: teams let the model decide control flow in prose instead of using explicit edges.
  • No recovery edge: errors happen, so the graph needs retry, fallback, or human-review paths.
  • Weak observability: without node-level traces you lose the main advantage of using a graph in the first place.

Hands-on exercise

Build a four-node LangGraph workflow for a document assistant: retrieve → analyze → review → respond. Add one failure edge that routes to human review if the answer is missing citations. Then inspect the state after each node to see how the graph makes reasoning visible.

Real-world use case

LangGraph is a strong fit for workflows where the same steps repeat: research agents, customer-support routing, verification pipelines, and developer assistants. Graphs make it easier to reason about loops, tool calls, and retries because the workflow stops being hidden inside the prompt.
