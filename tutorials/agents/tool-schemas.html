Tool Schema Design

Learn how to design agent tool schemas that are easy for the model to use, easy for your app to validate, and hard to misuse in production.

Objective & why schemas matter

In agent systems, tool schemas are the contract between language and action. A weak schema forces the model to guess arguments, while a strong schema constrains the model into a smaller and safer action space.

  • Clear parameter names reduce ambiguity.
  • Enums and required fields turn free-form text into validated actions.
  • Narrow tools are easier to monitor and safer to expose.
Python — Pydantic schema for a safe agent tool
from pydantic import BaseModel, Field
from typing import Literal

class TicketLookupInput(BaseModel):
    ticket_id: str = Field(description="Exact support ticket identifier")
    channel: Literal["email", "chat", "web"] = Field(description="Source channel for the ticket")
    include_history: bool = Field(default=False, description="Whether to attach recent ticket history")


def lookup_ticket(input: TicketLookupInput) -> dict:
    return {
        "ticket_id": input.ticket_id,
        "status": "open",
        "owner": "billing-ops",
        "history_included": input.include_history,
    }

Schema design checklist

  1. Name the tool after the action. lookup_ticket is clearer than process_data.
  2. Prefer concrete fields. Break “context” or “payload” into specific arguments.
  3. Use enums for bounded choices. Channels, priority levels, and statuses should not be unconstrained strings.
  4. Validate before execution. The tool should reject malformed input before touching external systems.
Python — reject invalid arguments before the tool runs
raw_args = {"ticket_id": "A-1042", "channel": "slack", "include_history": True}

try:
    parsed = TicketLookupInput.model_validate(raw_args)
    result = lookup_ticket(parsed)
except Exception as exc:
    result = {"error": f"Invalid tool input: {exc}"}

print(result)
💡 Design for correction

Good tool schemas do not just block bad calls. They make it obvious what the model should fix next. When a tool rejects an enum or a missing field, the agent can recover with a structured retry instead of guessing blindly.

Schema smells

  • One giant input blob: the model has too much freedom and validation becomes shallow.
  • Hidden side effects: the tool name sounds safe, but the implementation writes to external systems.
  • Arguments that leak policy: permission decisions should be enforced by the app, not delegated to free-form model text.
  • No audit data: the system cannot tell which exact arguments triggered the action.

Hands-on exercise

Take one broad tool from your current agent design and split it into two narrow tools with explicit schemas. Add at least one enum field and one required field. Then log the raw model arguments, the validated schema object, and the final tool result so you can compare how much ambiguity disappeared.

Real-world use case

In support, finance, and devtools agents, tool schemas are one of the main safety levers. When a model must choose from narrow validated arguments, the application retains control over what actions are even possible. That is often the difference between a reliable internal copilot and an unpredictable automation layer.
