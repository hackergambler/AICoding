Function Calling Basics

Learn Function Calling Basics in the AI Agents course. Practical Python code and real-world examples.

What function calling solves

Function calling gives an agent a safe bridge from language to software. Instead of pretending it knows the answer, the model can ask your application to run a specific function with structured arguments.

  • Good for search, databases, calculators, tickets, notifications, and workflow actions
  • Bad for arbitrary shell access, free-form SQL, or unlimited side effects without checks

1. The lifecycle of a tool call

Every function-calling system follows the same shape:

  1. You define the tool schema
  2. The model decides whether to call it
  3. Your code validates and executes the call
  4. You append the tool result
  5. The model uses that result to answer the user
Python - function calling lifecycle
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "Get current weather for a city.",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {"type": "string"},
                "units": {"type": "string", "enum": ["celsius", "fahrenheit"]}
            },
            "required": ["city"]
        }
    }
}]

2. Write schemas that are easy to validate

The model can only call tools as well as you define them. Strong schemas use explicit types, enums, and required fields. They avoid overloaded tools and vague descriptions.

  • Prefer city over location_text
  • Use enums for known choices like sort_order or units
  • Keep tool names action-oriented: search_orders, create_ticket, send_email
  • Return small structured payloads, not giant raw dumps
💡 Tool descriptions are prompt engineering

The model reads your schema text as instructions. A good description often fixes wrong tool selection faster than swapping models.

3. Validate before you execute

The model is proposing arguments, not granting itself authority. Your code still owns execution safety.

Python - validate arguments first
import json

def validate_get_weather(raw_args):
    args = json.loads(raw_args)
    city = args.get("city", "").strip()
    units = args.get("units", "celsius")

    if not city:
        raise ValueError("city is required")
    if units not in {"celsius", "fahrenheit"}:
        raise ValueError("invalid units")

    return {"city": city, "units": units}

4. Complete the loop correctly

Many beginners stop after the model requests a tool. The real pattern is two calls: first to get the tool request, second to let the model turn the result into a user-facing answer.

Python - complete execution loop
resp = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "What is the weather in Tokyo?"}],
    tools=tools,
    tool_choice="auto"
)

msg = resp.choices[0].message
tool_call = msg.tool_calls[0]
args = validate_get_weather(tool_call.function.arguments)
result = get_weather(**args)

messages = [
    {"role": "user", "content": "What is the weather in Tokyo?"},
    msg,
    {
        "role": "tool",
        "tool_call_id": tool_call.id,
        "content": json.dumps(result)
    }
]

final = client.chat.completions.create(model="gpt-4o-mini", messages=messages)
print(final.choices[0].message.content)

5. Security boundaries for real systems

As soon as tools can change data, send messages, or trigger workflows, you need boundaries:

  • Read-only tools and write tools should be separate
  • High-risk actions should require confirmation or a human check
  • Never pass raw shell commands or SQL directly from model output
  • Log every tool call with arguments and result previews
ℹ️ Safe progression

Build read-only tools first. Add state-changing tools only after the validation and logging path is solid.
