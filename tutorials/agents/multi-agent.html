Multi-Agent Systems

Build multi-agent systems with bounded roles, shared state, hand-off contracts, and evaluation loops instead of letting agents talk endlessly.

Why Use More Than One Agent?

Multiple agents help when one role benefits from a different prompt, tool set, or success metric than another. The point is not to create a conference room of bots. The point is to separate responsibilities so complex work becomes more inspectable and controllable.

Use multi-agent only when it earns its keep

If a single agent with good tools can do the job, prefer the simpler design. Multi-agent systems add state, latency, and failure points.

Three Reliable Patterns

1. Orchestrator → workers

A central planner breaks the task into sub-jobs and sends them to specialized workers. Great for research, extraction, and code-review style workflows.

2. Reviewer loop

One agent drafts, another critiques, and the first revises. Useful when quality control matters more than raw speed.

3. Router pattern

A small controller picks the best worker based on task type, input shape, or confidence.

Python - multi-agent hand-off contract
crew_state = {
    "goal": "Write a source-backed industry brief",
    "tasks": [],
    "artifacts": [],
    "open_questions": [],
    "final_status": "planning",
}

handoff = {
    "from": "searcher",
    "to": "synthesizer",
    "artifact_ids": ["src_14", "src_22"],
    "next_action": "draft_claims",
    "stop_if": ["citations missing", "sources older than 30 days"]
}

Shared State Design

The biggest hidden problem in multi-agent systems is not prompt quality. It is messy shared state. If each agent hands off a giant prose blob, the pipeline becomes impossible to debug. Use structured state: tasks, artifacts, claims, scores, and unresolved questions.

Bound the Conversation

  • Set a maximum number of turns per hand-off.
  • Require every worker to emit a structured artifact.
  • Give each worker a stop condition, not an open-ended chat.
  • Escalate when workers disagree instead of forcing endless debate.
Common failure pattern

Agents keep talking because nothing defines completion. Bounded turns and structured artifacts are what make multi-agent systems practical.

Where This Shows Up in Real Builds

  • Research Crew Agent — searcher, reader, synthesizer, editor.
  • Coding Agent — planner, executor, reviewer loops.
  • LangGraph Basics — stateful orchestration for repeatable control flow.
  • Agent Monitoring — observe hand-off quality and bottlenecks.

Evaluation Questions

Measure whether multi-agent actually helped:

  • Did answer quality improve compared with a strong single-agent baseline?
  • Where did latency increase the most?
  • Which hand-off produced the most rework or uncertainty?
  • Could any worker be collapsed back into code or a simpler prompt?

Practice Exercise

Build a tiny two-worker system: one agent extracts facts from a document, the second drafts a summary. Use a structured hand-off object. Then compare it against a single-agent baseline and note when the extra complexity actually helped.

Practice mission

Turn this lesson into a small but durable agent workflow. The goal is not just to reproduce the snippet, but to make the behaviour observable, bounded, and easy to debug after the first failure.

  1. Implement the lesson in a clean file or route with explicit input and output contracts.
  2. Add one logging line for latency, one for failures, and one for the final output or trace ID.
  3. Test the happy path plus two edge cases that would matter in a real Agents workflow.
  4. Write down what you would monitor before calling this production-ready.
💡 What “done” looks like

You should be able to explain why the approach works, show one working run, name the top failure mode, and point to the next place in the stack where you would add safeguards.

Production checklist

  • Validate and sanitize inputs before they reach the model or tool layer.
  • Keep logs or traces for the full run, not just the final answer.
  • Add at least one regression example you can rerun after refactors.
  • Document the fallback behaviour when the model, retriever, or tool fails.
  • Record the next metric to improve: latency, groundedness, cost, or reliability.
