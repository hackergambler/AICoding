OpenAI Function Calling

Learn the tool-calling loop, design strong schemas, validate arguments, and build reliable function-calling agents without hidden chaos.

1. Treat tool calling as a control loop

Function calling is not “the model magically runs code.” It is a loop with clear ownership boundaries:

  1. Your app sends the user request and the available tool schemas.
  2. The model decides whether a tool should be called and with what arguments.
  3. Your application validates those arguments and executes the tool.
  4. Your application sends the tool result back to the model.
  5. The model turns that result into the final answer.

The model decides what to call. Your backend decides whether the call is valid and safe enough to run.

Python — one tool schema
tools = [
    {
        "type": "function",
        "function": {
            "name": "lookup_ticket",
            "description": "Fetch a support ticket by ID.",
            "parameters": {
                "type": "object",
                "properties": {
                    "ticket_id": {"type": "string"},
                },
                "required": ["ticket_id"],
            },
        },
    }
]
Keep schemas boring

Short names, explicit required fields, and narrow enums beat giant all-purpose tools almost every time.

2. Design tools around real backend capabilities

A good tool maps cleanly to a real action in your system. A bad tool asks the model to improvise too much. Prefer small, composable tools:

  • lookup_order(order_id) instead of customer_support_everything(...)
  • search_docs(query, product) instead of one huge knowledge tool with dozens of optional fields
  • create_calendar_event(...) and update_calendar_event(...) as separate tools when semantics differ

The tighter your tool contract, the less you will debug strange arguments later.

3. Validate arguments before you execute anything

Never execute model-suggested arguments blindly. Validate required fields, enum values, numeric ranges, and authorization. Tool calling feels powerful because the model can propose actions. It becomes safe only when your backend behaves like a strict gatekeeper.

Python — safe dispatch layer
import json

ALLOWED_TOOLS = {"lookup_ticket": lambda ticket_id: {"ticket_id": ticket_id, "status": "open"}}

def dispatch_tool(tool_name: str, raw_arguments: str):
    if tool_name not in ALLOWED_TOOLS:
        raise ValueError(f"Unknown tool: {tool_name}")

    arguments = json.loads(raw_arguments)

    ticket_id = arguments.get("ticket_id")
    if not ticket_id or not isinstance(ticket_id, str):
        raise ValueError("ticket_id must be a non-empty string")

    return ALLOWED_TOOLS[tool_name](ticket_id=ticket_id)
Production rule

Validation belongs in your code, not in the prompt. Prompts reduce mistakes; validators prevent bad writes and unsafe side effects.

4. Run the full tool loop clearly

A robust function-calling app usually has an explicit loop rather than one giant helper. That loop makes logging, retries, and observability much easier.

Python — minimal function-calling loop
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=messages,
    tools=tools,
)

choice = response.choices[0].message
messages.append(choice.model_dump())

for call in choice.tool_calls or []:
    result = dispatch_tool(call.function.name, call.function.arguments)
    messages.append(
        {
            "role": "tool",
            "tool_call_id": call.id,
            "content": json.dumps(result),
        }
    )

final_response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=messages,
)

print(final_response.choices[0].message.content)

Notice the two important boundaries: the model does not execute the tool, and the final answer is generated only after the real tool result comes back.

5. Plan for the failures you will absolutely see

  • Wrong tool chosen: improve descriptions, reduce overlap, and remove ambiguous multi-purpose tools.
  • Arguments almost right but not valid: tighten parameter descriptions and add backend validation messages.
  • The tool succeeds but the final answer ignores the result: feed the result back in a compact and explicit structure.
  • The model loops too much: cap tool iterations and fall back to a human-readable error.
Python — iteration guard
MAX_TOOL_STEPS = 3

for step in range(MAX_TOOL_STEPS):
    # call model
    # execute tool calls
    # break when no tool calls remain
    pass
