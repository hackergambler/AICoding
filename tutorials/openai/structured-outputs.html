Structured Outputs with OpenAI

Design schema-safe OpenAI responses with JSON mode, typed validation, retries, and production-grade contracts.

In this lesson
  • Title & Objective
  • Concept Explanation
  • Code Example
  • Step-by-step breakdown
  • Hands-on Exercise

1. Title & Objective

Your goal is to make model output behave like an API response instead of a fragile paragraph. In production, structured output lets your frontend, tools, and databases consume model answers without guessing which sentence contains the real value. This lesson shows a schema-first workflow for OpenAI responses so you can ship extraction, routing, and summarization systems with much lower breakage.

ℹ️ Premium pattern

Think in contracts, not prompts. A great prompt helps quality, but a strong contract keeps systems reliable when responses become noisy or tasks become more complex.

2. Concept Explanation

Structured output usually fails for one of three reasons: the schema is too vague, the model is asked to do too much in one pass, or the application trusts malformed output without validating it. The fix is to split the system into a generation step and a verification step. The model proposes structured data; your app verifies and either accepts, repairs, or retries.

  • Use small, explicit field names.
  • Prefer enums or bounded values over free-form text whenever possible.
  • Tell the model what to do when information is missing instead of letting it invent.
  • Validate every response before downstream use.

3. Code Example

Python · OpenAI + Pydantic extraction contract
from openai import OpenAI
from pydantic import BaseModel, Field, ValidationError

class SupportTicket(BaseModel):
    priority: str = Field(description="low | medium | high | urgent")
    team: str = Field(description="billing | support | security | sales")
    summary: str
    needs_human: bool

client = OpenAI()

SYSTEM_PROMPT = """You classify inbound tickets.
Return only fields requested by the schema.
If a field is unknown, choose the safest valid option and explain uncertainty in summary.
"""

user_text = "Customer says their account was charged twice and they want a refund today."

response = client.responses.create(
    model="gpt-4.1-mini",
    input=[
        {"role": "system", "content": SYSTEM_PROMPT},
        {"role": "user", "content": user_text},
    ],
    text={
        "format": {
            "type": "json_schema",
            "name": "support_ticket",
            "schema": {
                "type": "object",
                "properties": {
                    "priority": {"type": "string", "enum": ["low", "medium", "high", "urgent"]},
                    "team": {"type": "string", "enum": ["billing", "support", "security", "sales"]},
                    "summary": {"type": "string"},
                    "needs_human": {"type": "boolean"}
                },
                "required": ["priority", "team", "summary", "needs_human"],
                "additionalProperties": False
            }
        }
    }
)

payload = response.output[0].content[0].text

try:
    ticket = SupportTicket.model_validate_json(payload)
    print(ticket.model_dump())
except ValidationError as exc:
    print("Validation failed:", exc)

4. Step-by-step breakdown

  1. Define the business contract. Here the application needs a team, priority, human-escalation flag, and short summary.
  2. Constrain the model. Enums reduce output drift and make downstream routing deterministic.
  3. Validate after generation. Even when the model is good, your app remains the source of truth.
  4. Repair only the minimum. If a field is missing, either retry with a focused repair prompt or send to a safe default workflow.
  5. Log rejected payloads. Those failures become your prompt-eval dataset.
💡 Practical rule

For extraction flows, shorter output schemas almost always outperform giant all-in-one JSON contracts. Split classification, extraction, and decisioning into separate calls if needed.

5. Hands-on Exercise

Build a schema for lead-qualification or customer-support intake. Add these fields: intent, sentiment, confidence, and recommended_action. Then:

  • Run five messy real-world examples.
  • Store the rejected payloads.
  • Add a second repair prompt that only fixes the invalid fields.
  • Compare one-pass acceptance rate before and after the repair step.

6. Real-world use case

Structured outputs are the backbone of AI ops dashboards, support triage, invoice extraction, moderation escalations, and agent tool routing. The biggest win is not prettier JSON — it is the ability to make downstream systems predictable.

7. Difficulty + Next Steps

Difficulty: Intermediate. You should already be comfortable with the OpenAI client and basic Python data models. Next, pair this with rate limits & retries so your structured workflows stay reliable under real traffic, then connect the results to Workflow Automation or data debugging.
