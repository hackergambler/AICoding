Chat Completions Deep Dive

Learn how to structure messages, design strong system prompts, manage conversation state, and ship reliable Chat Completions flows.

📑 In This Lesson
  • 1. Understanding the Messages Array
  • 2. The Three Roles Explained
  • 3. Crafting Effective System Prompts
  • 4. Building Conversation History
  • 5. Key Parameters
  • 6. Practice Exercise

1. Understand the request lifecycle

Chat Completions works best when you think of every request as a complete snapshot of the conversation so far. The model does not remember earlier API calls unless you include the relevant turns again. Your job is to decide what context belongs in the window, what should be summarized, and what can be left out.

A strong first mental model is: system instructions define behavior, user messages define the task, and assistant messages preserve prior reasoning or prior answers that matter for continuity. If the model sounds inconsistent, the issue is often missing or bloated context rather than the model itself.

Python — minimal chat completion request
from openai import OpenAI

client = OpenAI()

messages = [
    {"role": "system", "content": "You are a concise backend mentor. Explain with one code sample and one warning."},
    {"role": "user", "content": "How do I read JSON from a file in Python?"}
]

response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=messages,
    temperature=0.2,
    max_tokens=300,
)

print(response.choices[0].message.content)
print(response.usage)
Why beginners get unstable answers

They keep changing the hidden rules. If the system message, output format, and examples drift from call to call, the assistant will also drift.

2. Use the three roles intentionally

Each role has a distinct job. When you mix them up, the model loses signal.

  • system: permanent behavior, tone, safety boundaries, output contract, and non-negotiable rules.
  • user: the current request, the source material, and any constraints supplied by the human.
  • assistant: prior replies that are worth keeping because they preserve continuity or a previous structured answer.

Do not dump logs, documents, and tool payloads directly into a system prompt. Put large task inputs into the user turn. Keep the system prompt compact and durable.

Python — structured role design
messages = [
    {
        "role": "system",
        "content": (
            "You are a code reviewer. Always respond with: "
            "1) risk summary, 2) exact fix, 3) quick test plan."
        ),
    },
    {
        "role": "user",
        "content": (
            "Review this FastAPI endpoint and tell me where validation is weak.

"
            "<paste endpoint code here>"
        ),
    },
]

3. Write a system prompt the model can actually follow

Strong system prompts are short, specific, and testable. They describe role, audience, output shape, and hard constraints. They do not try to encode the entire product spec in one giant paragraph.

  • Start with the assistant identity: You are a debugging copilot for Python web apps.
  • Add the audience: Assume the reader is a junior engineer.
  • Add the response contract: Return three bullets, one code block, and one verification step.
  • Add hard boundaries: Do not invent environment variables or package names.
System prompt skeleton you can reuse
system_prompt = """
You are an AI coding tutor for backend developers.
Audience: engineers who know Python basics but are new to API design.
Response contract:
- Start with a 2 sentence explanation.
- Then show one working code snippet.
- End with one likely mistake and how to test the fix.
Constraints:
- Be concrete.
- Do not mention tools the user did not ask about.
- Do not invent file paths or env vars.
""".strip()
A useful test

Ask the same question three times with slightly different user wording. If the answer shape changes wildly, your system prompt is still too vague.

4. Manage conversation history like a product engineer

In real apps you cannot keep appending messages forever. Cost rises, latency rises, and the model starts paying attention to stale context. Good chat products trim, summarize, and pin only what matters.

A practical strategy is to keep:

  • the system prompt,
  • the latest task turn,
  • the last few user/assistant exchanges,
  • and a summary of older context when the thread grows long.
Python — trim long histories safely
def build_history(system_prompt: str, recent_turns: list[dict], summary: str | None = None) -> list[dict]:
    messages = [{"role": "system", "content": system_prompt}]

    if summary:
        messages.append(
            {
                "role": "system",
                "content": f"Conversation summary so far: {summary}",
            }
        )

    messages.extend(recent_turns[-8:])
    return messages

Store raw history in your database, but send a carefully chosen subset to the model. That distinction is what makes production chat systems feel coherent instead of expensive and noisy.

5. Tune the parameters that actually matter

Most teams over-focus on exotic settings and under-focus on three fundamentals:

  • temperature: lower for deterministic coding help, higher for brainstorming.
  • max_tokens: cap output so a runaway answer does not burn budget.
  • message quality: the clearest lever of all. Bad prompts create bad answers no matter how you tune temperature.
Python — preflight a coding request
def create_code_reply(client, messages: list[dict]):
    return client.chat.completions.create(
        model="gpt-4o-mini",
        messages=messages,
        temperature=0.1,
        max_tokens=500,
    )
Parameter rule of thumb

For code generation, start at temperature 0 to 0.2. Raise it only when you deliberately want alternative approaches or naming ideas.

6. Turn it into a real feature

A solid practice project is a tiny coding copilot with persistent chat history, history trimming, and a repeatable answer shape. Build it in four steps:

  1. Create a message builder that separates system prompt, recent turns, and summary memory.
  2. Log usage and latency on every call.
  3. Add a guardrail that rejects empty prompts and overlong pasted payloads.
  4. Add a follow-up route that can continue the same conversation safely.

Once that works, your next improvements are not more prompt tricks. They are observability, caching, and better context selection.

  • Continue with streaming responses
  • Move into tool and function calling
  • Debug slow or broken response streaming
  • Fix auth and key failures
  • Build a real assistant project
