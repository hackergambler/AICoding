OpenAI Embeddings

Learn how to generate embeddings, choose chunking and metadata strategy, build semantic retrieval, and debug weak search quality.

1. What embeddings are actually for

An embedding turns text into a numeric vector so you can compare meaning rather than exact wording. That is why embeddings power semantic search, deduplication, clustering, and retrieval pipelines. They do not replace a language model. They help you decide which information the language model should read next.

The quality of an embedding system depends on more than the model. Chunk boundaries, metadata, normalization, filtering, and evaluation matter just as much.

Python — generate one embedding and inspect the shape
from openai import OpenAI

client = OpenAI()

response = client.embeddings.create(
    model="text-embedding-3-small",
    input="Rate limiting prevents a client from overloading your API.",
)

vector = response.data[0].embedding
print(len(vector))
print(vector[:5])
Key mental model

Embeddings answer “what is similar to this text?” They do not answer the user directly. A retrieval layer still needs ranking, filtering, and answer generation.

2. Build the text before you build the vector

Poor source text creates poor embeddings. Before you vectorize anything, decide how documents will be chunked and what metadata each chunk must carry. A strong chunk usually contains:

  • one coherent idea,
  • enough surrounding context to stand on its own,
  • metadata like source, section, title, and timestamp,
  • and a stable document or chunk identifier.

If your chunks are too large, the vectors become vague. If they are too small, they lose context and retrieval becomes fragmentary.

Python — prepare documents before embedding
def prepare_chunk(text: str, *, source: str, section: str, chunk_id: str) -> dict:
    cleaned = " ".join(text.split())
    return {
        "id": chunk_id,
        "text": cleaned,
        "metadata": {
            "source": source,
            "section": section,
        },
    }

3. Generate vectors in batches and store metadata beside them

Batch requests are cheaper to manage and easier to retry. More importantly, never store vectors without metadata. Retrieval quality depends on what you can filter and explain later.

Python — batch embedding pipeline
def embed_texts(client, texts: list[str]):
    response = client.embeddings.create(
        model="text-embedding-3-small",
        input=texts,
    )
    return [row.embedding for row in response.data]

chunks = [
    "Rate limiting protects APIs from bursty traffic.",
    "Caching reduces repeated database reads.",
    "Vector search finds semantically similar chunks.",
]

vectors = embed_texts(client, chunks)
print(len(vectors), len(vectors[0]))

When you write into a vector store, persist both the vector and the descriptive metadata. You will need that metadata later for filtering, debugging, and citation quality.

4. Retrieval quality comes from ranking, not just cosine similarity

A minimal semantic search flow looks like this:

  1. Embed the query.
  2. Fetch the top k nearest chunks.
  3. Filter by metadata where needed.
  4. Optionally rerank or compress the candidates.
  5. Only then send the best chunks into your answer model.
Python — cosine similarity from scratch
import math

def cosine_similarity(a: list[float], b: list[float]) -> float:
    dot = sum(x * y for x, y in zip(a, b))
    mag_a = math.sqrt(sum(x * x for x in a))
    mag_b = math.sqrt(sum(y * y for y in b))
    return dot / (mag_a * mag_b)
Why good chunks still retrieve badly

The query itself may be too vague. Try query rewriting, metadata filters, or returning more candidates before final ranking.

5. Evaluate the retrieval layer before you trust it

Do not evaluate embeddings only by “the app feels decent.” Create a small gold set of user questions and the chunks that should appear for each one. Then measure whether your pipeline actually retrieves them.

  • Track hit rate at top 3 or top 5.
  • Inspect false positives and ask why they looked similar.
  • Try chunk-size experiments one variable at a time.
  • Keep difficult queries as regression tests.
Python — retrieval evaluation sketch
gold = {
    "How do I secure an API key?": ["security-01", "env-vars-02"],
    "What chunk size should I use for docs?": ["rag-05", "rag-06"],
}

def hit_at_k(retrieved_ids: list[str], relevant_ids: list[str], k: int = 5) -> int:
    return int(any(doc_id in retrieved_ids[:k] for doc_id in relevant_ids))

6. Common failure patterns and the next route

  • Duplicate chunks dominate results: deduplicate at ingest time.
  • All results are vaguely related: your chunks are too broad or your query rewrite is weak.
  • Correct results exist but rank too low: increase k, filter better, or add a reranking step.
  • Answers hallucinate anyway: retrieval helped, but the generation prompt still needs citations and answer constraints.

Once embeddings work, the next leap in quality comes from chunk design and evaluation discipline.

  • Tune chunking with text splitting
  • Add keyword + vector hybrid search
  • Debug vector database failures
  • Fix data quality and ingestion bugs
  • Use retrieval inside a real project
