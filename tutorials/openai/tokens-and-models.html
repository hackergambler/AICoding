Tokens, Models & Cost Optimization

Understand tokens, context budgeting, model routing, and how to control latency and cost before prompt size becomes a production problem.

1. Tokens are your real budget unit

In language-model systems, the true resource is not characters or words. It is tokens. Tokens influence cost, latency, and whether the model can even fit your prompt into its context window. This is why prompt design, retrieval chunking, and chat history trimming all connect back to token budgeting.

Python — estimate tokens with tiktoken
import tiktoken


def count_tokens(text: str, model: str = "gpt-4o-mini") -> int:
    encoding = tiktoken.encoding_for_model(model)
    return len(encoding.encode(text))

sample = "Summarize this pull request in three bullets."
print(count_tokens(sample))
Practical rule

Count tokens before you send big prompts. Teams often debug “model quality” when the real issue is that the useful context never fit in the request.

2. Budget the window before you build the feature

A request competes for space between multiple components:

  • system instructions,
  • conversation history,
  • retrieved documents,
  • tool results,
  • and the response you still want the model to generate.

If you do not reserve room for the answer, the output gets clipped or the request fails. Build a budget explicitly instead of stuffing everything into the window and hoping for the best.

Python — reserve output budget
def prompt_budget(total_window: int, expected_output: int, safety_margin: int = 500) -> int:
    return total_window - expected_output - safety_margin

usable_input_tokens = prompt_budget(total_window=128000, expected_output=1500)
print(usable_input_tokens)

3. Route tasks to the smallest model that can succeed

Do not send every request to the most capable model. Create a routing strategy based on task type:

  • small model for classification, short rewrites, tagging, and guardrails,
  • mid-tier model for normal chat and coding help,
  • larger model only when reasoning depth or synthesis quality really matters.

This single change often gives better cost control than any prompt tweak.

Python — simple model router
def choose_model(task: str) -> str:
    if task in {"classification", "rewrite", "extract"}:
        return "gpt-4o-mini"
    if task in {"chat", "code_help", "retrieval_answer"}:
        return "gpt-4o"
    return "gpt-4o"

4. Prevent silent truncation and prompt bloat

Three problems show up repeatedly in production:

  1. The conversation history keeps growing forever.
  2. RAG pipelines inject too many chunks.
  3. Tool outputs are pasted back in verbatim instead of summarized.

Every one of those problems steals space from the part the user actually cares about.

Python — trim retrieved context by budget
def take_chunks_with_budget(chunks: list[str], max_tokens: int, model: str) -> list[str]:
    chosen, used = [], 0
    for chunk in chunks:
        size = count_tokens(chunk, model=model)
        if used + size > max_tokens:
            break
        chosen.append(chunk)
        used += size
    return chosen
Good habit

Summarize tool payloads and older conversation turns into compact state. Raw payload replay is one of the fastest ways to waste a context window.

5. Measure latency and usage as part of quality

A beautiful answer that takes too long or costs too much is still a bad product decision. Log model name, prompt tokens, completion tokens, and latency on every request. Then inspect which features are consuming the budget.

Python — log usage after each call
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=messages,
)

print({
    "model": response.model,
    "prompt_tokens": response.usage.prompt_tokens,
    "completion_tokens": response.usage.completion_tokens,
    "total_tokens": response.usage.total_tokens,
})

6. Next routes for better control

Once you understand tokens and routing, you can make stronger decisions about retrieval, streaming, and caching.

  • Stream long answers to reduce perceived latency
  • Use embeddings for retrieval instead of stuffing everything into context
  • Study caching for repeated prompts
  • Debug latency and network bottlenecks
  • Fix history and memory bloat
