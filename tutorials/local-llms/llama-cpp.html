llama.cpp — Run LLMs Without GPU

Use llama.cpp to run quantized LLMs on CPU-only machines. Server mode, Python bindings, and OpenAI-compatible API.

What llama.cpp is really good at

llama.cpp is the simplest path from "I want to run a model on my own machine" to a real local inference server. It is excellent when you want CPU-first inference, quantized GGUF models, low operational overhead, and a predictable deployment surface.

  • Best for local assistants, offline tools, edge devices, and small internal services
  • Input format GGUF model files
  • Typical workflow pick model → choose quantization → run CLI or server → call from app
ℹ️ Think in memory first

With local LLMs, model size is only half the story. Your real constraint is RAM or VRAM after quantization plus enough headroom for context and concurrent requests.

1. Choose the right model and quantization

The biggest beginner mistake is downloading a model too large for the machine. Start with a smaller instruct model and a moderate quantization such as Q4_K_M. It is usually the best balance between quality and memory use.

  • 3B–8B Q4 — practical for laptops and CPU-only experimentation
  • 7B–14B Q4/Q5 — better quality if you have enough RAM
  • Q8 — higher quality, much more memory
  • Q2/Q3 — small footprint, noticeable quality drop
💡 Safe first pick

If you are unsure, start with a 3B or 7B instruct model in Q4_K_M. Learn the tooling first. Only then move up in size.

2. Build and run llama.cpp from source

bash - compile and run a GGUF model
# Clone and build
git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp
make -j4

# Run a quantized instruct model
./llama-cli \
  -m ./models/Llama-3.2-3B-Instruct-Q4_K_M.gguf \
  -p "Write a Python function to reverse a string." \
  --ctx-size 4096 \
  --n-predict 256

If the CLI works, you have a known-good foundation. Do this before you wrap it in Python, LangChain, a web UI, or an agent loop.

3. Turn it into an OpenAI-compatible local server

The server mode is where llama.cpp becomes genuinely useful for applications. Once it exposes an OpenAI-style endpoint, many SDKs and internal tools can talk to it without major code changes.

bash - start local inference server
./llama-server \
  -m ./models/Llama-3.2-3B-Instruct-Q4_K_M.gguf \
  --host 0.0.0.0 \
  --port 8080 \
  --ctx-size 4096 \
  -np 4
Python - call llama-server with the OpenAI SDK
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8080/v1",
    api_key="not-needed"
)

resp = client.chat.completions.create(
    model="local-3b",
    messages=[{"role": "user", "content": "Summarize what a Python decorator does."}],
    temperature=0.2
)

print(resp.choices[0].message.content)

4. Tune the four flags that matter most

You can ignore many flags at the beginning. These four usually matter most:

  • --ctx-size — maximum context window; larger uses more memory
  • -t / n_threads — CPU threads; too high can reduce efficiency
  • -ngl / n_gpu_layers — number of layers offloaded to GPU, if available
  • -np — number of parallel requests; useful for light internal services
Python - benchmark core settings
from llama_cpp import Llama

llm = Llama(
    model_path="./models/model.gguf",
    n_ctx=4096,
    n_threads=8,
    n_gpu_layers=0,
    verbose=False
)

out = llm.create_chat_completion(
    messages=[{"role": "user", "content": "Explain Python generators simply."}],
    max_tokens=180
)

print(out["usage"])

5. Common local-LLM failure patterns

  1. Model will not start — wrong GGUF file, corrupted download, or not enough memory.
  2. Responses are extremely slow — model too large, threads misconfigured, or context window too high.
  3. Server works but app fails — wrong local URL, SDK pointed at cloud instead of local, or incompatible request shape.
  4. Output quality is poor — over-quantized model, weak system prompt, or the task really needs a stronger model.

Use deployment debugging when the local server runs but your app cannot reach it. Continue to HuggingFace Local when you want more model flexibility, and to Fine-tuning Local LLMs when prompting alone is not enough.

6. Best next build

Once you can run one local model reliably, wrap it in a small application. A good next step is a private internal helper or offline personal assistant with a narrow task and a clear prompt. The strongest follow-on build here is Personal Assistant.
