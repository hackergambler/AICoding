LangChain with Local LLMs

Use LangChain with Ollama for local chat, retrieval, and structured workflows without losing control of latency or debugging visibility.

1. Know when LangChain helps and when it is extra weight

LangChain is useful when you want composable prompts, retrievers, chains, structured outputs, and interchangeable model backends. It is less useful when your app only needs one direct model call. Use it when orchestration complexity is real, not just because it is popular.

Healthy rule

First prove the model works through raw Ollama. Then add LangChain so you know whether later bugs come from the runtime or the orchestration layer.

2. Start with a clean local chat chain

Python — minimal LangChain + Ollama chain
from langchain_ollama import ChatOllama
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

llm = ChatOllama(model="llama3.1:8b", temperature=0.2)

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a concise backend mentor."),
    ("human", "{question}"),
])

chain = prompt | llm | StrOutputParser()
print(chain.invoke({"question": "Explain Python generators with one example."}))

This pattern matters because it separates prompt design, model invocation, and output parsing. That separation becomes very valuable once you add retrieval or structured responses.

3. Add local embeddings and retrieval without leaving the machine

One of the strongest reasons to use LangChain with Ollama is to keep both generation and embeddings local. That gives you privacy and a consistent local workflow for RAG experiments.

Python — local embeddings for retrieval
from langchain_ollama import OllamaEmbeddings

embeddings = OllamaEmbeddings(model="nomic-embed-text")
vector = embeddings.embed_query("How does rate limiting work?")
print(len(vector))
Python — retrieval outline
# Split docs -> embed locally -> store vectors -> retrieve top chunks
# Then feed chunks into a prompt chain

Keep the retrieval step observable. Log what chunks were selected, their metadata, and the final prompt length. Otherwise local RAG failures feel mysterious.

4. Use structured outputs and small components

When you build larger local workflows, resist the temptation to create giant chains. Compose small parts:

  • prompt formatting,
  • retrieval,
  • model call,
  • parser or validator,
  • and optional fallback logic.

Smaller components make it easier to swap local models, inspect intermediate state, and debug bad outputs.

Python — add streaming for local UX
for chunk in chain.stream({"question": "Explain async IO in simple terms."}):
    print(chunk, end="", flush=True)

5. Debug the local orchestration layer explicitly

  • Chain works in code but feels slow: measure whether the bottleneck is retrieval, model generation, or parsing.
  • RAG answers ignore context: inspect retrieved chunks before blaming the model.
  • Outputs are inconsistent: shrink the prompt and simplify the parser.
  • Model works in Ollama CLI but not in LangChain: compare raw runtime settings and stop token behavior.
A practical habit

Keep one raw Ollama test script beside the LangChain version. It gives you a fast control case whenever orchestration starts behaving strangely.
