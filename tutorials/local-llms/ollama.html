Run LLMs Locally with Ollama

Use Ollama as a practical local runtime, serve models through its API, tune generation, and connect it cleanly to retrieval and application code.

1. Think of Ollama as your local model server

Ollama is valuable because it turns local models into a simple runtime you can script against. Instead of managing every tokenizer and checkpoint detail yourself, you get a consistent command-line and HTTP interface for experimentation and app building.

bash — install and pull a first model
# macOS / Linux
curl -fsSL https://ollama.com/install.sh | sh

ollama --version
ollama pull llama3.2
ollama run llama3.2
Use the CLI first

Before wiring a UI or API layer, prove the model answers correctly in the CLI. That makes later debugging much faster.

2. Pick models intentionally

Ollama makes model swapping easy, which is helpful but also dangerous. Do not compare models casually after changing three other variables at the same time. Keep a benchmark prompt set and compare quality, latency, and memory one variable at a time.

  • small model for fast prototyping,
  • mid-size model for balanced local assistants,
  • specialized model only when your task clearly benefits from it.

3. Use the HTTP API like a real app backend

Ollama runs a local server, which makes it easy to integrate with Python, JavaScript, or a desktop app.

Python — call Ollama over HTTP
import requests

payload = {
    "model": "llama3.2",
    "prompt": "Explain caching in a web API in three bullets.",
    "stream": False,
}

response = requests.post("http://localhost:11434/api/generate", json=payload, timeout=60)
response.raise_for_status()
print(response.json()["response"])
Python — stream tokens from Ollama
with requests.post(
    "http://localhost:11434/api/generate",
    json={"model": "llama3.2", "prompt": "Write a short release note.", "stream": True},
    stream=True,
    timeout=60,
) as response:
    response.raise_for_status()
    for line in response.iter_lines():
        if line:
            print(line.decode("utf-8"))

4. Use Modelfiles for repeatable local behavior

One of the most practical Ollama features is the Modelfile. It lets you define a base model, system prompt, and runtime defaults so your local assistant behaves consistently across sessions.

Modelfile — define a reusable local assistant
FROM llama3.2
SYSTEM You are a concise AI coding mentor. Always give one working example.
PARAMETER temperature 0.2

That pattern is especially useful for internal tools, demos, and offline workflows where you want stable behavior without rebuilding prompts in multiple files.

5. Connect Ollama to retrieval and app logic carefully

Once the runtime is stable, you can add embeddings, retrieval, or a small orchestration layer. The mistake to avoid is adding every advanced feature at once. Prove each layer in order:

  1. direct prompt works,
  2. API integration works,
  3. streaming works,
  4. retrieval works,
  5. state and memory work.
Good engineering sequence

If your local app fails, strip it back to a raw /api/generate call first. Only reintroduce retrieval, memory, and chaining once the runtime layer is stable.

6. Common Ollama failures and next routes

  • Connection refused: the Ollama service is not running.
  • Slow generation: the model is too large or the machine has fallen back to CPU-heavy execution.
  • Inconsistent outputs: compare prompt, temperature, and model choice before blaming the runtime.
  • Local RAG feels weak: inspect chunking and retrieval before changing the model.
  • Add LangChain orchestration on top of Ollama
  • Compare Ollama with raw Transformers loading
  • Connect Ollama to a RAG workflow
  • Fix runtime and deployment issues
  • Use Ollama inside a real local project
