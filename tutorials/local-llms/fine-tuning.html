Fine-tuning Local LLMs with LoRA

Fine-tune open-source LLMs with LoRA/QLoRA using the Unsloth library. Train on custom data in minutes.

When fine-tuning is the right move

Fine-tuning is powerful, but it should come after you try better prompting, retrieval, and cleaner task design. The best reason to fine-tune is not "I want the model to know more facts." It is usually one of these:

  • You need a stable output format every time
  • You want a specific house style or domain tone
  • You need the model to follow repeated task patterns more reliably
  • You want a smaller model to perform better on your narrow task
ℹ️ Fine-tuning is not a shortcut for missing knowledge

If the task depends on frequently changing or long reference material, RAG is usually the better answer. Fine-tuning stores behavior and patterns better than it stores fresh facts.

1. LoRA vs QLoRA vs full fine-tuning

For most practitioners, the decision is simple:

  • LoRA — lightweight adapters on the base model
  • QLoRA — LoRA on a quantized base model; lower VRAM, great default
  • Full fine-tuning — rare for small teams; expensive and operationally heavier

In this project, QLoRA is the best practical default because it gets you most of the gains without requiring a large GPU budget.

2. Build a good training dataset

Data quality matters far more than the number of examples. A few hundred clean examples with consistent formatting will usually beat thousands of messy ones.

jsonl - instruction tuning format
{"instruction": "Explain a Python decorator in plain English.", "input": "", "output": "A decorator is a function that wraps another function to add behavior without changing the original function body."}
{"instruction": "Convert this bug report into a structured ticket.", "input": "Checkout page throws 500 when card expires.", "output": "severity: high\ncomponent: payments\nsummary: checkout returns 500 on expired card"}
  • Keep task framing consistent
  • Remove contradictory examples
  • Make the target outputs look exactly like production output
  • Reserve a validation split from the start

3. Train with Unsloth on a realistic budget

bash - install Unsloth stack
pip install unsloth transformers datasets trl accelerate bitsandbytes
Python - QLoRA starter run
from datasets import load_dataset
from unsloth import FastLanguageModel
from trl import SFTTrainer

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="unsloth/llama-3.2-3b-bnb-4bit",
    max_seq_length=2048,
    load_in_4bit=True
)

model = FastLanguageModel.get_peft_model(
    model,
    r=16,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],
    lora_alpha=16,
    lora_dropout=0
)

dataset = load_dataset("json", data_files="train.jsonl", split="train")

trainer = SFTTrainer(
    model=model,
    tokenizer=tokenizer,
    train_dataset=dataset,
    dataset_text_field="text",
    max_seq_length=2048
)

trainer.train()
model.save_pretrained("./adapters/my-task-lora")
💡 Start tiny on purpose

Run a first experiment with a few hundred examples and 1–2 epochs. Your goal is to verify the training loop and measure behavior change, not to maximize benchmark scores on day one.

4. Evaluate before you celebrate

A model that sounds better is not always better. Compare the base model and the tuned model on a held-out set with the same prompt structure. Evaluate format adherence, task accuracy, refusal behavior, and hallucinations.

Python - simple before/after evaluation loop
eval_cases = [
    "Convert this bug report into a structured incident record.",
    "Write a concise retail support reply for a delayed order."
]

for case in eval_cases:
    print("CASE:", case)
    print("BASE:", run_base(case))
    print("TUNED:", run_tuned(case))
    print("-" * 80)

For retrieval-heavy tasks, add a RAG baseline too. Sometimes the tuned model loses because retrieval would have solved the problem more cleanly.

5. Merge, serve, and monitor the adapter

After training, decide how you want to serve the result:

  • Keep adapters separate and load them at runtime
  • Merge adapters into the base weights for simpler deployment
  • Export to a format that fits your local stack, including a path back into llama.cpp or HuggingFace local inference
ℹ️ Production mindset

Your tuned model now becomes a software dependency. Version the dataset, track the base model, log evaluation results, and store the exact training configuration so you can reproduce wins later.

6. Failure patterns to expect

  1. The model parrots training examples — dataset too narrow or over-trained.
  2. Format adherence improved but facts got worse — fine-tuning fixed style, not knowledge.
  3. Training crashes — VRAM limits, wrong sequence length, or incompatible package versions.
  4. The tuned model is slower or harder to serve — deployment plan was ignored during training.

Use configuration debugging for environment issues and continue to Dev Agent when you want to embed fine-tuned behavior into a real system.
