Claude Computer Use Workflows

Design safe browser and desktop automation loops with Claude, screenshot feedback, approval gates, and bounded action plans.

In this lesson
  • Title & Objective
  • Concept Explanation
  • Code Example
  • Step-by-step breakdown
  • Hands-on Exercise
  • Real-world use case
  • Difficulty + Next Steps

1. Title & Objective

This lesson shows how to build a Claude-powered computer-use workflow that can inspect a UI, propose actions, request approval when needed, and keep a trace of every step. The objective is not raw automation speed; it is bounded automation that remains understandable.

2. Concept Explanation

Computer use becomes dangerous when the agent has too much freedom and too little memory of what just happened. A good loop separates observation, planning, execution, and verification. Claude should never act on a stale screenshot or continue indefinitely without guardrails.

  • Observation: fresh screenshot + current goal + last action result.
  • Plan: single next action, not a giant multi-step dream.
  • Execution: tool layer performs one click, type, scroll, or navigate.
  • Verification: new screenshot confirms whether the step succeeded.

3. Code Example

Python · bounded browser action loop
from anthropic import Anthropic

client = Anthropic()

TOOLS = [
    {"name": "click", "description": "Click a named target on screen"},
    {"name": "type_text", "description": "Type text into a focused field"},
    {"name": "scroll", "description": "Scroll the viewport up or down"},
]

messages = [{
    "role": "user",
    "content": "Goal: log into the admin console and open billing settings. Use one action per turn. Stop if credentials are requested."
}]

response = client.messages.create(
    model="claude-3-7-sonnet-latest",
    max_tokens=700,
    tools=TOOLS,
    messages=messages,
)

# Your runner should inspect the tool request, require approval if sensitive,
# execute one action, attach a new screenshot, and call Claude again.

4. Step-by-step breakdown

  1. Start with a goal and a screenshot. Without state, the model guesses.
  2. Allow exactly one high-confidence action per turn.
  3. Tag sensitive actions such as credential entry, payments, destructive updates, and downloads.
  4. Require approval for sensitive actions and stop if the UI state changes unexpectedly.
  5. Store every observation, tool call, and screenshot hash for replay.
⚠️ Safety rule

Never allow unrestricted typing, unrestricted navigation, or unbounded loops on privileged systems. Computer-use agents need the same change-management thinking as deployment scripts.

5. Hands-on Exercise

Build a sandboxed browser runner that opens a documentation site, searches for a page, and captures a quote. Add approval on any navigation outside the original domain. Keep a replay log of every action and screenshot.

6. Real-world use case

Computer-use workflows shine in internal back-office tasks, QA walkthroughs, form filling, and legacy dashboards that lack APIs. They become risky when teams treat them like autonomous humans instead of controlled step runners.

7. Difficulty + Next Steps

Difficulty: Advanced. After this, study Playwright agents, sandbox execution, and the Dev Agent capstone.
