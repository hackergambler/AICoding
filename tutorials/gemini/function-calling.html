Gemini Function Calling

Give Gemini access to real-world tools: define functions, parse tool calls, execute code, and build complete agentic workflows.

Why function calling matters

Function calling turns Gemini from a text generator into a decision maker. Instead of guessing the weather, searching your product catalog from memory, or pretending it saved a file, the model can request real code execution and wait for your application to send the result back.

  • Model decides which tool to call and with what arguments
  • Your code validates those arguments before anything runs
  • Your tool executes the side effect or lookup
  • The result returns to Gemini for the final grounded answer
ℹ️ Use tools for facts and actions

Good tool calls are for fresh data, private data, or side effects: weather, tickets, orders, SQL, files, calendar, email, and internal workflows. If the answer is already in the prompt, a normal completion is simpler.

1. Design tools with strict, boring schemas

The best tool definitions are boring. They avoid ambiguity, keep field names obvious, and make validation easy. Gemini does better when your tools look like clean APIs, not like clever one-off helpers.

Python - explicit tool schema
import google.generativeai as genai

weather_tool = {
    "name": "get_weather",
    "description": "Get the latest weather for a city.",
    "parameters": {
        "type": "object",
        "properties": {
            "city": {"type": "string", "description": "City name"},
            "units": {"type": "string", "enum": ["celsius", "fahrenheit"]}
        },
        "required": ["city"]
    }
}

Prefer small tools that do one clear thing. A tool called search_orders is easier to reason about than a giant business_helper tool with twenty optional fields.

2. Give Gemini a clear execution contract

Tell the model exactly when to use tools, when not to use them, and how to behave after a tool result comes back. This is where many "function calling is unreliable" complaints actually begin.

Python - system instruction for predictable tool use
system_instruction = """You are an operations assistant.
Use tools whenever the user asks for live data, account-specific data, or a workflow action.
Do not invent tool results.
If a required parameter is missing, ask a short follow-up question.
After receiving a tool result, answer in plain English and mention only the useful fields.
"""

model = genai.GenerativeModel(
    model_name="gemini-1.5-flash",
    tools=[weather_tool],
    system_instruction=system_instruction
)

3. Run the loop: model → tool → model

The full pattern is always two-stage. Gemini first decides whether to call a tool. Your app executes it. Then Gemini uses the returned data to produce the final answer.

Python - minimal Gemini tool loop
def get_weather(city, units="celsius"):
    return {"city": city, "temp": 27, "condition": "Sunny", "units": units}

chat = model.start_chat()
first = chat.send_message("What is the weather in Hyderabad right now?")

part = first.candidates[0].content.parts[0]
args = dict(part.function_call.args)
result = get_weather(**args)

second = chat.send_message({
    "function_response": {
        "name": part.function_call.name,
        "response": result
    }
})

print(second.text)
💡 Log every tool call in development

Store the tool name, arguments, validation result, latency, and output size. Tool systems feel magical until they fail. Logs are what make them fixable.

4. Validate arguments before execution

Never trust model-generated arguments blindly. Gemini is good, but it is still a probabilistic system. Your application should reject invalid values, clamp limits, and sanitize anything that touches a database, shell, or third-party API.

  • Check types and required fields
  • Whitelist enums such as units, status, or sort order
  • Set safe bounds like limit <= 20
  • Reject raw SQL, file paths, or arbitrary code strings
Python - simple validation guard
def validate_weather_args(args):
    city = args.get("city", "").strip()
    units = args.get("units", "celsius")

    if not city:
        raise ValueError("city is required")
    if units not in {"celsius", "fahrenheit"}:
        raise ValueError("units must be celsius or fahrenheit")

    return {"city": city, "units": units}

5. Multi-tool workflows need routing, not chaos

Once you add more than one tool, start thinking in routes: lookup tools, reasoning tools, and action tools. Do not give every tool every permission. A common production split is:

  • Read-only tools — search, fetch, retrieve, summarize
  • State-changing tools — create ticket, send email, update record
  • High-risk tools — payments, account changes, destructive actions

Use confirmations or human review before executing high-risk tools. This lesson connects naturally with Function Calling Basics and the broader AI Agents course.

6. Common Gemini tool-use failures

  1. The model answers without calling a tool — your system instruction is too weak or the question sounds answerable from general knowledge.
  2. Wrong argument names — your schema and your execution function drifted apart.
  3. Tool loops forever — no max step limit or no stopping condition after the result returns.
  4. Bad downstream action — missing validation or missing permission boundaries.

For failures around environment or runtime behavior, route into configuration debugging or network debugging. For more Gemini-specific capabilities, continue to Advanced Features and Gemini Embeddings.

7. Build the next real project

After one working tool loop, the strongest next step is a small workflow project, not a giant agent. Try a single operational assistant that can answer account questions and perform one action safely. The best follow-on build here is Workflow Automation.
