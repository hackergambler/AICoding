Gemini Multi-turn Chat

Build Gemini chat systems that keep useful context, trim bad history, and stay controllable as conversations grow longer.

Objective & what “chat” really means

A chat model is not a magical memory engine. A Gemini chat session is still an application pattern: you decide what history to keep, what to summarize, what to redact, and what belongs in durable product memory versus short-lived conversation state.

  • Conversation history helps continuity, but can also carry mistakes forward.
  • Summaries compress old turns into a smaller state representation.
  • App memory stores durable facts such as user preferences or ticket IDs outside the raw chat log.
Python — application-managed Gemini chat with rolling summary
from google import genai
from google.genai import types
import os

client = genai.Client(api_key=os.environ["GEMINI_API_KEY"])
MODEL = "gemini-1.5-flash"

history = []
summary = ""


def ask(question: str) -> str:
    global summary, history

    if len(history) > 8:
        summary_response = client.models.generate_content(
            model=MODEL,
            contents=[
                "Summarize the conversation for future turns in under 120 words.",
                *history
            ],
            config=types.GenerateContentConfig(temperature=0.1)
        )
        summary = summary_response.text
        history = history[-4:]

    prompt_parts = []
    if summary:
        prompt_parts.append(f"Conversation summary: {summary}")
    prompt_parts.extend(history)
    prompt_parts.append(question)

    response = client.models.generate_content(
        model=MODEL,
        contents=prompt_parts,
        config=types.GenerateContentConfig(temperature=0.4)
    )

    history.append(question)
    history.append(response.text)
    return response.text

Step-by-step breakdown

  1. Separate durable memory from chat history. User profile, account status, and saved preferences should live in your application store, not only in prior messages.
  2. Keep a rolling window. Raw transcripts grow quickly and increase cost. Trim aggressively after the critical context is summarized.
  3. Protect system instructions. The assistant role and product rules should be re-applied every turn, not inferred from prior chat behavior.
  4. Log conversation state transitions. When a chat assistant behaves strangely, you need to know which summary or prior turn distorted the answer.
💡 Strong chat products use layered memory

Short-term history helps the current turn. Long-term memory belongs in your app database. Retrieval belongs in a document system. Mixing all three into one growing prompt is the fastest way to make a chat app expensive and unreliable.

Common chat failure modes

  • History pollution: one incorrect assistant answer becomes part of the context and steers later turns.
  • Token drift: chat quality degrades because important earlier turns are buried under irrelevant chatter.
  • Memory confusion: the system acts as if previous messages are durable product facts.
  • No reset or escalation path: users need a way to restart, summarize, or hand off the conversation.
Python — durable facts belong in app state, not transcript only
user_profile = {
    "name": "Asha",
    "plan": "pro",
    "preferred_language": "English",
    "team": "support-ops"
}

system_context = f"""
You are the support copilot.
Always answer in {user_profile['preferred_language']}.
Respect the current plan: {user_profile['plan']}.
Use the account facts supplied by the application as ground truth.
"""

Hands-on exercise

Build a customer-success chat assistant with three layers of state:

  • a rolling transcript for the last few turns
  • a summary field for older context
  • an application record for durable facts such as customer ID, plan, and recent ticket status

Then test what happens when the transcript contains a mistaken answer. Does the durable application state override it?

Real-world use case

Internal copilots, support agents, and coaching assistants all need stateful interaction. The best products do not simply “remember everything.” They keep only the context that improves the next turn, compress old discussion, and retrieve durable facts from reliable storage when needed.
