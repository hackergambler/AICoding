Text Splitting Strategies

Choose chunk size and overlap by document type, inspect chunks before indexing, and turn text splitting into an evaluated engineering decision.

1. Splitting is one of the highest-leverage decisions in RAG

Chunking decides what the retriever is allowed to find. If chunks are too broad, the system retrieves vague evidence. If they are too small, important relationships get broken apart. Great RAG apps often look “smart” because the chunking strategy is disciplined.

Rule before rules of thumb

Do not memorize one universal chunk size. Split based on how your users ask questions and how your source documents are structured.

2. Choose chunking strategy by content type

  • API docs: smaller chunks, often around headings, endpoints, parameters, and examples.
  • tutorials and guides: medium chunks that keep explanation and code together.
  • narrative or policy documents: larger chunks can preserve argument flow better.
  • tables or structured records: split by logical rows or sections, not just character count.

The right chunking strategy follows the document’s structure instead of flattening everything into uniform slices.

Python — recursive splitter baseline
from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,
    chunk_overlap=200,
    length_function=len,
)

chunks = splitter.split_text(long_text)
print(len(chunks))
print(chunks[0][:200])

3. Use token-aware thinking, not only character counts

Character counts are a practical approximation, but what really matters is how much context fits in the answer stage later. If your retrieval system returns five chunks that each look reasonable in characters but explode in token count, your answer prompt will still bloat.

Python — inspect chunk token sizes
import tiktoken

enc = tiktoken.get_encoding("cl100k_base")

def token_len(text: str) -> int:
    return len(enc.encode(text))

for chunk in chunks[:5]:
    print(token_len(chunk))

That is why good chunking is always connected to prompt budgeting and answer-generation design.

4. Keep metadata and hierarchy when you split

When a document is split into chunks, preserve the context around each chunk:

  • document title,
  • section heading,
  • page or file path,
  • and a stable chunk index.

This makes retrieval more interpretable and helps you rebuild a larger parent section when needed.

Python — attach chunk metadata
chunk_records = []

for idx, chunk in enumerate(chunks):
    chunk_records.append(
        {
            "id": f"doc-12-chunk-{idx}",
            "text": chunk,
            "metadata": {
                "title": "Caching Guide",
                "section": "Invalidation",
                "chunk_index": idx,
            },
        }
    )

5. Evaluate chunking with real retrieval tests

Do not trust chunking because it “looks sensible.” Create a small query set and compare outcomes across chunk sizes and overlap settings.

  • Measure whether the relevant chunk appears in the top results.
  • Inspect whether the chunk is complete enough to answer the question.
  • Look for duplicate or near-duplicate retrievals.
  • Track how many tokens the final answer prompt consumes.
Experiment grid — one variable at a time
experiments = [
    {"chunk_size": 500, "overlap": 100},
    {"chunk_size": 800, "overlap": 120},
    {"chunk_size": 1200, "overlap": 200},
]
What “good” looks like

The best chunking setup is the one that retrieves complete, specific evidence while keeping the final prompt small enough for the answer model to use well.
