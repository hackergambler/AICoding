RAG Evaluation: RAGAS, Faithfulness & Answer Relevancy

Learn how to evaluate a RAG pipeline end to end instead of guessing whether retrieval, chunking, or generation is the real problem.

In This Lesson

  1. Why RAG Evaluation Matters
  2. The Four Core RAGAS Metrics
  3. Running RAGAS Evaluation
  4. Evaluating Without Ground Truth
  5. Tracing with LangSmith
  6. Hands-on Exercise

Objective & evaluation layers

RAG quality is not one number. A bad answer can come from the retriever, the chunker, the prompt, the model, or the evaluation set itself. A strong evaluation stack separates these concerns so teams know what broke instead of only seeing “the answer looked wrong.”

  • Retrieval metrics check whether the right evidence was fetched.
  • Groundedness metrics check whether the answer stays supported by the retrieved context.
  • Task success metrics check whether the answer was actually useful for the user outcome.
Python — simple offline RAG evaluation harness
from dataclasses import dataclass
from typing import List

@dataclass
class EvalCase:
    question: str
    gold_doc_ids: List[str]
    generated_answer: str
    retrieved_doc_ids: List[str]
    retrieved_context: str


def precision_at_k(retrieved_ids, gold_ids, k=5):
    top_k = retrieved_ids[:k]
    hits = sum(1 for doc_id in top_k if doc_id in gold_ids)
    return hits / max(len(top_k), 1)


def recall_at_k(retrieved_ids, gold_ids, k=5):
    top_k = set(retrieved_ids[:k])
    return len(top_k.intersection(set(gold_ids))) / max(len(gold_ids), 1)


def grounded_answer(case: EvalCase) -> bool:
    answer = case.generated_answer.lower()
    context = case.retrieved_context.lower()
    key_terms = [term for term in answer.split() if len(term) > 5]
    return sum(term in context for term in key_terms) >= 3

The four core RAGAS metrics

Track faithfulness, answer relevance, context precision, and context recall separately. Each one points to a different failure layer: grounding, generation quality, retrieval noise, or missing evidence.

How to build a useful eval set

  1. Start with real user questions. Synthetic queries can help bootstrap, but production RAG fails on the messy questions users actually ask.
  2. Label supporting sources. At least for a seed set, identify the document or chunk IDs that should support the answer.
  3. Create error buckets. Tag failures as retrieval miss, chunking issue, unsupported answer, stale document, or poor final synthesis.
  4. Compare by slice. Long questions, acronym-heavy questions, and multi-document questions usually fail differently.
Python — record failure buckets so you can improve the right layer
from collections import Counter

failures = Counter()

for case in eval_cases:
    if recall_at_k(case.retrieved_doc_ids, case.gold_doc_ids, k=5) == 0:
        failures["retrieval_miss"] += 1
    elif not grounded_answer(case):
        failures["unsupported_answer"] += 1
    else:
        failures["other"] += 1

print(failures)
💡 Measure before prompt-tuning

Teams often spend days changing prompts when the real issue is chunking or retrieval. If your gold document is never retrieved, no answer prompt will fix the system.

Evaluating without ground truth

When you do not have labelled answers yet, use synthetic evals, human review, and trace-based heuristics. This is still better than shipping blind and hoping the answer “looks right”.

Running RAGAS evaluation

Start with a small offline eval set and automate it in CI. Even 25–50 carefully chosen examples can reveal regressions much faster than manual spot-checking.

Common evaluation mistakes

  • Only judging final answers: you lose visibility into which subsystem failed.
  • No negative queries: good eval sets include “I do not know” cases, not only answerable questions.
  • Too little slice coverage: a tiny eval set can hide failures on code docs, policy docs, tables, or long documents.
  • Manual review without structure: reviewers need a rubric or results become noisy and hard to compare across experiments.

Tracing with LangSmith

Evaluation results are far more useful when they link back to traces. Review the exact retrieved chunks, tool calls, and prompts that produced a low-scoring answer.

Hands-on exercise

Create a 30-question eval set for one knowledge base. Label the supporting sources for at least 15 of the questions. Compare two chunking strategies and two retrieval strategies. Report:

  • precision@5 and recall@5
  • unsupported-answer rate
  • the top three failure buckets you observed

Real-world use case

In a policy assistant or internal docs assistant, the business risk is not just a wrong answer. The risk is a confident answer not supported by the approved documents. A good evaluation framework shows whether the system found the right policy, quoted it accurately, and stayed grounded under pressure.
