Hybrid Search: BM25 + Vector Retrieval

Combine keyword and embedding retrieval the right way so your RAG system stops missing exact terms while still understanding semantic intent.

Objective & why hybrid search matters

Dense retrieval is great at semantic similarity, but it can miss rare tokens, SKUs, legal clause names, or product IDs. Lexical retrieval is excellent at exact matches, but often misses paraphrases. Hybrid search combines both so the system can recover precise evidence and broader meaning.

  • Lexical retrieval catches exact words, abbreviations, and high-signal terms.
  • Dense retrieval catches paraphrases and conceptually related passages.
  • Fusion merges the candidate lists into one ranked result set.
Python — hybrid retrieval with BM25 + embeddings
from rank_bm25 import BM25Okapi
from sentence_transformers import SentenceTransformer
from sklearn.metrics.pairwise import cosine_similarity
import numpy as np

chunks = [
    "The invoice dispute workflow starts in the billing queue.",
    "Account lockouts should be escalated to identity support.",
    "Refunds for duplicate charges require finance approval.",
]

tokenized = [chunk.lower().split() for chunk in chunks]
bm25 = BM25Okapi(tokenized)
encoder = SentenceTransformer("all-MiniLM-L6-v2")
chunk_vectors = encoder.encode(chunks, normalize_embeddings=True)


def hybrid_search(query: str, k: int = 3):
    bm25_scores = bm25.get_scores(query.lower().split())
    query_vec = encoder.encode([query], normalize_embeddings=True)
    dense_scores = cosine_similarity(query_vec, chunk_vectors)[0]

    bm25_norm = (bm25_scores - bm25_scores.min()) / (bm25_scores.ptp() + 1e-8)
    dense_norm = (dense_scores - dense_scores.min()) / (dense_scores.ptp() + 1e-8)
    fused = 0.45 * bm25_norm + 0.55 * dense_norm

    ranked_ids = np.argsort(fused)[::-1][:k]
    return [(chunks[i], float(fused[i])) for i in ranked_ids]

print(hybrid_search("How do duplicate-charge refunds get handled?"))

How score fusion works

  1. Generate two candidate sets. Run lexical and dense retrieval independently.
  2. Normalize the score spaces. BM25 and cosine scores are not directly comparable.
  3. Fuse with a simple weighted sum or reciprocal-rank fusion. Start simple before inventing complex rankers.
  4. Evaluate by query type. Product-ID queries and conceptual questions often need different weights.
💡 A practical default

When you do not yet have strong evaluation data, use hybrid search as the safer starting point for enterprise RAG. Exact terminology matters too much in support, compliance, finance, and internal documentation to rely on dense retrieval alone.

Failure patterns

  • No normalization: one scorer dominates simply because its scale is larger.
  • Bad chunking: fusion cannot rescue passages that were split in ways that destroy meaning.
  • No query analysis: all queries are treated the same, even when some are clearly keyword-heavy.
  • Only top-1 evaluation: hybrid search is often valuable because it improves candidate recall before reranking.

Hands-on exercise

Take 25 real questions from your docs or product knowledge base. Label each as keyword-heavy, semantic, or mixed. Compare dense-only, BM25-only, and hybrid retrieval using precision@5. Then test whether a reranker on top of hybrid beats the simple fusion baseline.

Real-world use case

An internal support assistant often needs to retrieve answers containing exact names like billing codes, escalation queues, product SKUs, and environment variables. Hybrid search works well because those exact tokens matter, but users still phrase their questions naturally. That is exactly the gap fusion closes.
