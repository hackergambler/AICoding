System Prompts — Persona, Rules & Constraints

Master system prompts: set personas, enforce output formats, add guardrails, and create consistent AI assistants.

What system prompts actually control

A system prompt is the operating policy for the model inside your application. It should define role, tone, rules, output shape, and what the model should do when information is missing or the request is unsafe. Good systems feel consistent because the prompt gives them a stable center of gravity.

  • Role — what the assistant is and is not
  • Intent — what outcomes matter most
  • Constraints — rules, safety, boundaries, and refusal behavior
  • Execution — how the response should be shaped

1. Start with a strong prompt skeleton

Python - durable system prompt skeleton
SYSTEM = """
You are MYRQ Support AI, a senior product support assistant.

Goals:
- Resolve the user's issue accurately.
- Ask at most one clarifying question when critical information is missing.
- Prefer practical steps over theory.

Rules:
- Never invent account data or tool results.
- If policy or facts are missing, say what is unknown.
- Use short sections and bullet points when the answer has multiple steps.

Output contract:
- Start with the direct answer.
- Then give steps.
- End with one next action.
"""

This kind of structure is more reliable than a vague instruction like "be helpful and professional." Specific beats poetic almost every time.

2. Use the RICE framework on every serious prompt

RICE is a practical checklist for strong system prompts:

  • Role — who the assistant is
  • Intent — what success looks like
  • Constraints — what not to do
  • Execution — how the response should be shaped
Python - RICE example for code review
SYSTEM = """You are CodeReview-AI, a senior Python reviewer.

Role:
- Review Python code for correctness, readability, and security.

Intent:
- Find the most important issues first.
- Prefer fixes that are easy to apply.

Constraints:
- Do not rewrite entire files unless necessary.
- Do not invent runtime behavior you cannot infer from the code.

Execution:
- Return sections named: Risks, Fixes, Cleaned Snippet, Follow-up Check.
"""

3. Separate instructions from user data clearly

Many prompting failures come from mixing instructions and raw user content together. Use delimiters and explicit labels so the model knows what is policy and what is just input to analyze.

Python - delimiters reduce confusion
messages = [
    {"role": "system", "content": SYSTEM},
    {
        "role": "user",
        "content": "Analyze the report inside <report> tags and summarize only the root cause.\n<report>...raw text here...</report>"
    }
]
ℹ️ Why delimiters help

They reduce accidental instruction bleed. The model can more reliably distinguish the application's rules from the user's data or pasted documents.

4. Add an output contract, not just a persona

Personas are useful, but output contracts are what make systems feel stable in production. Tell the model the sections you want, whether JSON is required, and how uncertainty should be expressed.

System prompt fragment - enforce response shape
Return your answer in this structure:
1. Direct answer
2. Key reasons
3. Recommended next step
4. Risks or unknowns

If you are uncertain, say "Unknown" rather than guessing.

Once you need reusable examples, combine this lesson with Few-Shot Prompting. When you need structured reasoning, continue to Chain-of-Thought.

5. Guardrails belong in the system prompt too

The system prompt should describe what the assistant must refuse, what requires confirmation, and when it should ask a follow-up question instead of making assumptions.

  • Refuse unsafe or restricted actions
  • Do not invent tool or database results
  • Ask for missing identifiers before acting on an account
  • Prefer concise clarifying questions over long speculative answers
💡 Good refusal rules sound calm, not dramatic

Write refusal behavior the same way you write product requirements: plainly, specifically, and without overexplaining. The goal is consistent behavior, not theatrical safety language.

6. Evaluate system prompts like product code

Do not trust a prompt because it "feels good." Test it against a small suite of realistic cases: straightforward requests, ambiguous cases, refusal cases, long inputs, and formatting checks.

Python - tiny prompt evaluation harness
tests = [
    "Reset my password and keep the reply under 5 bullet points.",
    "Summarize this policy and note anything unknown.",
    "Use account 12345 and issue a refund immediately."
]

for prompt in tests:
    answer = run_with_system(prompt)
    print("PROMPT:", prompt)
    print(answer)
    print("-" * 60)

7. The most common system-prompt mistakes

  1. Too vague — strong tone, weak behavior rules
  2. Too long and repetitive — bloated instructions that fight themselves
  3. No output contract — responses vary wildly from request to request
  4. No uncertainty policy — the model guesses instead of admitting what it does not know
