Project: Coding Agent

Build a production-minded coding agent that reads a repository, plans safe patches, runs tests in a sandbox, and reports exactly what changed.

What You Will Build

A capstone coding agent that can inspect a repository, choose the smallest useful patch, edit only the files it needs, run checks in a sandbox, and return a structured change report. Think of it as a disciplined engineering assistant rather than a magical autonomous coder.

Why this capstone matters

If you can build a coding agent that respects repo boundaries, budgets tokens, handles failing tests, and explains its edits, you understand a huge amount of modern agent engineering: tool schemas, sandboxes, planning, evals, and observability.

Capstone Architecture

1. Repository mapper

Read file trees, ignore generated folders, and create a compact repo map the model can reason over without dumping the whole codebase into context.

2. Task planner

Convert the user request into a patch plan with target files, risk notes, acceptance tests, and an explicit stop condition.

3. Editing tools

Use read, write, search, and apply-patch style tools with strict path validation so the agent cannot wander outside the workspace.

4. Sandbox runner

Execute tests and formatters in a constrained subprocess, then feed back only the relevant traces instead of the entire terminal log.

Python - repository map and patch-plan contract
from pathlib import Path
import json

ALLOWED_EXTS = {".py", ".js", ".ts", ".md", ".json"}
SKIP_DIRS = {".git", "node_modules", ".next", "dist", "build", "venv"}

def build_repo_map(root: str) -> list[dict]:
    entries = []
    for path in Path(root).rglob("*"):
        if path.is_dir():
            continue
        if any(part in SKIP_DIRS for part in path.parts):
            continue
        if path.suffix not in ALLOWED_EXTS:
            continue
        rel = path.relative_to(root).as_posix()
        entries.append({"path": rel, "size": path.stat().st_size})
    return entries

PATCH_PLAN_SCHEMA = {
    "goal": "string",
    "files_to_edit": ["src/app.py"],
    "tests_to_run": ["pytest tests/test_app.py -q"],
    "risk_notes": ["Do not rename public functions"],
    "done_when": ["All targeted tests pass", "No syntax errors"]
}

print(json.dumps({"repo_map": build_repo_map("."), "schema": PATCH_PLAN_SCHEMA}, indent=2))

Build Sequence

Milestone 1: Safe read-only understanding

Start with a read-only pass. Let the agent inspect the tree, open the top candidate files, and produce a patch plan before it writes anything. This is where most bad coding agents fail: they edit before they understand.

Milestone 2: Minimal patch generation

Require the model to propose the smallest viable change. If the task needs one function edit, do not let it rewrite five files. Small diffs are easier to test, review, and roll back.

Milestone 3: Test loop

After the first patch, run the narrowest test command possible, summarize the error, and let the agent attempt one focused repair. Keep the loop bounded with max attempts, timeouts, and explicit abort rules.

Python - bounded edit / test loop
MAX_ATTEMPTS = 3

for attempt in range(1, MAX_ATTEMPTS + 1):
    plan = plan_patch(task, repo_map, previous_errors)
    apply_patch(plan["patch"])
    test_result = run_command(plan["tests_to_run"][0], timeout=90)

    if test_result["returncode"] == 0:
        return {"status": "passed", "attempt": attempt, "summary": test_result["stdout"]}

    previous_errors = summarize_failure(test_result)

return {"status": "needs-human-review", "errors": previous_errors}

Guardrails That Make This Production-Friendly

  • Path allow-list: never let the agent read or write outside the workspace root.
  • Command policy: allow lint/test/build commands, block package installs, network fetches, and destructive shell actions by default.
  • Patch size caps: reject edits beyond a safe line threshold unless a human approves.
  • Human review gates: if public APIs, migrations, or auth code change, escalate before applying.
Common failure mode

The agent passes a narrow test but quietly breaks adjacent behavior. That is why your acceptance contract needs both targeted tests and one broader smoke test before you accept the patch.

How to Evaluate the Agent

Do not evaluate only on “did it eventually pass.” Track:

  • first-pass success rate
  • average patch size
  • attempts per successful task
  • percentage of tasks requiring human escalation
  • regressions introduced after a green check

Final Output Checklist

  1. The agent can map a repo without overflowing context or reading forbidden paths.
  2. Every patch comes with files touched, tests run, and a done-when contract.
  3. All commands execute inside a bounded sandbox with timeouts and logs.
  4. The system has a human-escalation path for risky or ambiguous changes.
  5. You can explain why the agent changed each file, not just that it changed them.
