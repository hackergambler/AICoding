Project: Research Crew Agent

Build a research crew that splits work across specialist agents for search, retrieval, synthesis, citation checking, and final editorial review.

What You Will Build

A research crew that behaves like a small editorial team. One agent gathers candidate sources, another extracts evidence, a synthesis agent drafts claims, and a review agent checks citations, contradictions, and unsupported statements before publication.

Capstone mindset

A serious research system is not just “search plus summary.” It is a pipeline for claim handling: gather, compare, score confidence, draft, verify, and only then publish.

Team Roles and Hand-offs

Searcher

Find candidate documents and rank them by relevance, freshness, and authority.

Reader

Extract the few passages needed for the current question instead of dumping full articles into context.

Synthesizer

Draft a structured answer with explicit claims, supporting snippets, and uncertainty markers.

Editor / Fact checker

Reject unsupported claims, force citation coverage, and request another search round when evidence is weak.

Python - simple research crew state object
from dataclasses import dataclass, field
from typing import List, Dict

@dataclass
class Claim:
    text: str
    evidence_ids: List[str] = field(default_factory=list)
    confidence: str = "medium"

@dataclass
class CrewState:
    question: str
    sources: List[Dict] = field(default_factory=list)
    claims: List[Claim] = field(default_factory=list)
    draft: str = ""
    open_questions: List[str] = field(default_factory=list)
    status: str = "searching"

Build Sequence

Milestone 1: Retrieval discipline

Start with a predictable search-and-read loop. The crew should collect only the top few sources, normalize them into a shared state object, and store metadata like title, URL, date, and authority score.

Milestone 2: Claim drafting

The synthesis agent should not write a polished article first. Instead, have it draft a list of claims with evidence references. That makes editorial review much easier.

Milestone 3: Editorial correction

Before you generate the final report, run an editor agent that asks three questions for every major point: Do we have evidence? Are there conflicting sources? Is the date still current?

Python - citation coverage check
def validate_claims(claims, source_index):
    problems = []
    for claim in claims:
        if not claim.evidence_ids:
            problems.append({"claim": claim.text, "issue": "missing_citation"})
            continue
        missing = [sid for sid in claim.evidence_ids if sid not in source_index]
        if missing:
            problems.append({"claim": claim.text, "issue": "unknown_source", "sources": missing})
    return problems

RAG and Crew Design Decisions

  • Use retrieval before multi-agent complexity: most research failures come from poor evidence handling, not lack of more agents.
  • Keep shared state explicit: agents should pass structured fields like claims, citations, and open questions, not giant prose blobs.
  • Separate draft quality from evidence quality: a beautiful answer with weak citations is still a bad system.
Common failure mode

Research crews often amplify each other’s mistakes. If the first retrieval step is weak, the later agents sound confident while staying wrong. Add an explicit “search again” branch.

Final Output Checklist

  1. Every major claim in the final report has a citation path.
  2. The crew can request a new search round when evidence is weak or stale.
  3. The final answer separates confirmed facts, likely inferences, and unresolved questions.
  4. You can inspect each hand-off instead of treating the whole pipeline as a black box.
  5. The report is publishable because the evidence trail is strong, not only because the prose sounds good.
